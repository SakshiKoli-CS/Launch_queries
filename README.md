- [Apex→www redirect not working → apex must be configured in Launch first](custom-domains-ssl/launch-apex-to-www-redirect-requires-apex-configured-custom-hostnames.md) — the apex (naked) domain redirect can't take effect until the **apex is added + verified under Custom Hostnames**; then set up the **apex→www redirect in the Launch UI**. Fix/prevent: **add the apex in Launch** + **repoint the apex A record** (Fastly `151.101.66.137` → Launch `162.251.205.14`) so Launch manages both certs (apex now supported; no downtime). Also: an **encrypted private key** — ask for unencrypted key or password (redact both). _Keywords: cert renewed apex still expired, .bank FTLD, apex redirect Fastly old cert, add apex in Launch, update A record, encrypted private key password protected, move apex off Fastly._

## edge-functions-rewrites/ — Edge Functions, Edge URL Rewrites / launch.json

- [Custom 404 on a static (Nuxt SSG) site → Edge Function re-serves /404.html with 404 status](edge-functions-rewrites/launch-custom-404-static-nuxt-ssg-edge-function.md) — for SSG, Launch is a **static CDN** and shows its **system 404 before framework logic**, so a dropped `404.html` isn't served correctly. Fix: an **Edge Function** that intercepts upstream 404s and re-serves `/404.html` with explicit **HTTP 404** (gate on `Accept: text/html`, pass through `/_nuxt/` + `_payload.json`, `no-store`); works for nested routes, no `launch.json` rewrite. Alt: **SSR mode** (native error handling, needs a server runtime). _Keywords: custom 404 static SSG, Nuxt fully static, default not found page, 404.html not served, edge function 404 fallback, correct 404 status SEO, nested routes, SSR vs static._
- [CFL-0001 Internal Server Error → edge function failed (parent-domain routing)](edge-functions-rewrites/launch-cfl-0001-edge-function-error-academy-parent-domain-routing.md) — CFL-0001 = the **Edge Function failed** (unhandled exception/invalid response). "Our sub-app has no edge function" → it's served under a **parent domain** (e.g. `/academy` under `www.contentstack.com`) whose edge function the request flows through — debug the **parent's** edge function. _Keywords: CFL-0001, internal server error, edge function 500, no edge function in codebase, parent domain routing, academy docs dotcom, try-catch logging._
- [Next.js `after()` / `waitUntil()` works locally, not on Launch → serverless reaps the process](edge-functions-rewrites/launch-nextjs-after-waituntil-background-processing-serverless.md) — post-response background work in an API route isn't guaranteed on serverless (process killed after the response). Reliable options: **Launch Edge Functions support `waitUntil()`**, or **offload to a separate endpoint via `fetch(..., { keepalive: true })`**. _Keywords: Next.js after() waitUntil() background processing, works locally not Launch, serverless process killed after response, keepalive fetch, edge functions waitUntil supported, async background task._
- [Edge Function timeout / long-running (3–4 min LLM) → async pattern](edge-functions-rewrites/launch-edge-function-timeout-long-running-llm-async-pattern.md) — default **120 s**, raisable to **5 min** (cost approval, not instant); but a multi-minute **blocking** edge function isn't best practice. For "call-and-wait" work (524 on long LLM calls) use an **async pattern** (accept → process async → poll/callback) or offload to origin. _Keywords: edge function timeout, 120s, 5 min max, 524 LLM, long-running, call and wait, async, polling, offload to origin._
- [Edge Functions in TypeScript](edge-functions-rewrites/edge-functions-typescript.md) — Edge Functions run **JavaScript only**; author in TS, **compile to JS** into the `functions/` folder (sample `tsconfig.json` included). _Keywords: edge function, TypeScript, JavaScript only, tsconfig, build to functions folder._
- [Edge Functions: TS support + WinterCG runtime differences](edge-functions-rewrites/launch-edge-functions-typescript-support-wintercg-runtime.md) — JS only (compile TS→JS into `functions/`; codebase can stay TS); edge runtime is **lighter than Node.js** and **WinterCG-compliant** — don't assume full Node APIs at the edge (use Cloud Functions for heavier Node logic). _Keywords: Edge Functions TypeScript, WinterCG, edge runtime vs Node, lighter runtime, compile TS to JS, cloud functions._
- [Edge function rewrites/redirects broken after deploy → skipped (no valid export)](edge-functions-rewrites/launch-edge-function-skipped-no-valid-export-rewrites-redirects-broken.md) — deploy log "Step 5: Edge Functions Deployment — `[proxy].edge.js does not contain a valid export function. Skipping`" = function lacks a **valid default export**, silently not deployed (images 404, redirects don't fire). Fix the `export default function handler(...)`; check the deploy log's edge step. _Keywords: edge function not triggering, not deployed, valid export function, [proxy].edge.js, skipping deploying to edge, rewrites redirects broken, 404._
- [Subdomain→subdomain redirect + Edge Function vs Next.js middleware](edge-functions-rewrites/launch-subdomain-redirect-edge-function-vs-nextjs-middleware.md) — custom-domain UI redirect is **apex→subdomain only**; use an **Edge Function** (302) for subdomain→subdomain. Edge Functions beat Next.js middleware — **Launch doesn't run Next.js middleware at the edge** (server-side only). _Keywords: 302 redirect, apex to subdomain only, Edge Function redirect, Next.js middleware not on edge, personalization, pending domain DCV._
- [Rewrite/proxy works but URL bar switches to origin domain](edge-functions-rewrites/launch-rewrite-proxy-url-bar-changes-origin-absolute-urls.md) — the **origin (WordPress) emits absolute URLs** with its own domain (links/redirects/canonical); Launch forwards unchanged so the browser follows them. - [Phased migration: rewrite some paths to Launch, rest to legacy (Umbraco)](edge-functions-rewrites/launch-phased-migration-rewrites-legacy-backend.md) — point the **primary domain at Launch**, use **`launch.json` rewrites / Edge Functions** to serve migrated paths from Launch+Contentstack and forward the rest to the legacy origin. - [Environment redirects slow with ~300 configured → mechanism + lookup efficiency](edge-functions-rewrites/launch-environment-redirects-slow-large-count-performance.md) — count alone isn't the issue; **how matches are evaluated** is.
