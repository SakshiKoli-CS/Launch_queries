# Launch Support Knowledge Base

Markdown KB entries distilled from Contentstack **Launch** support conversations, organized by **feature area**. Each entry follows the QUESTION / ANSWER pattern (see `.cursor/rules/support-pattern.mdc`).

**How to use this index:** scan for the symptom or keyword that matches the query, then open the linked entry. Authoring/extraction rules and folder placement live in [`AGENTS.md`](AGENTS.md) and the support-pattern rule.

## custom-domains-ssl/ — custom domains, apex/A records, DCV, SSL/TLS

- ⭐ **[CANONICAL: contentstackapps.com Dangerous-Site/Malware flag — root cause, RCA, fix](custom-domains-ssl/launch-contentstackapps-dangerous-site-malware-canonical-rca-fix.md)** — Google **false-positive** flag on the shared default domain (triggered by `{brand}-dev` resembling the real brand site), cascaded to all `*.contentstackapps.com`. Fix: verify domain in **Google Search Console** (TXT via Cloudflare) → **Request Review** → add **noindex/robots.txt**. **[Official doc](https://www.contentstack.com/docs/launch/troubleshooting-browser-dangerous-site-warnings)**. Supersedes the per-thread entries below. _Keywords: contentstackapps.com flagged, dangerous site, malware, Google Search Console, false positive, noindex robots, RCA, troubleshooting doc._
- [Custom domain & SSL not active; no A record for apex](custom-domains-ssl/launch-custom-domain-apex-no-a-record-ssl-not-active.md) — apex must be **created in the Launch UI first**; A record + cert are only generated after that. _Keywords: custom domain inactive, no A record, SSL not active, apex domain._
- [Custom domain stuck in "Pending Validation" / DCV records](custom-domains-ssl/launch-custom-domain-pending-validation-dcv-records.md) — verify **DCV + TXT records** at DNS provider; stuck-but-correct domains can clear on upstream (Cloudflare) delay. _Keywords: pending validation, DCV, certificate stuck, TXT record._
- ["Coming Soon" page DNS + project/environment reuse](custom-domains-ssl/launch-coming-soon-page-dns-project-environment-reuse.md) — Launch **doesn't hand out DNS records**; configure a custom domain in the UI and add the **auto-generated** records. Use a **new environment** (not a new project) for a temporary page to save a project slot; repurpose later by redeploying + updating the domain. _Keywords: coming soon, DNS records, project quota, reuse project, new environment, custom domain per environment._
- [Default-domain Malware flag, multi-customer (Chrome + Firefox) + RCA](custom-domains-ssl/launch-default-domain-malware-flag-multi-customer-rca.md) — shared `*.contentstackapps.com` flagged by Google (via Search Console) across **multiple customers at once**; here **both Chrome AND Firefox** flagged it (so "Firefox works" isn't proof it's clean). RCA available; cleared platform-side; use custom domains. _Keywords: malware flag, multi-customer, Chrome Firefox, Google Search Console, RCA, contentstackapps.com._
- [Default-domain "Dangerous Site" flag: assets blocked + noindex/robots prevention](custom-domains-ssl/launch-contentstackapps-dangerous-site-noindex-robots-assets-blocked.md) — new `*.contentstackapps.com` sites flagged by Google Safe Browsing (Chrome/Safari warn, Firefox doesn't); flagged domains can also have **assets blocked**. Prevent via `noindex`/`X-Robots-Tag` + **robots.txt blocking Google's scanner**; use a custom domain for production. _Keywords: dangerous site, assets not rendering, noindex, robots.txt, Google Safe Browsing, contentstackapps.com._
- [Default-domain "Dangerous Site" (Edge works); custom domains unaffected](custom-domains-ssl/launch-default-domain-dangerous-site-edge-works-custom-domain-unaffected.md) — Chrome warns / **Edge works** = Google Safe Browsing flag on the default domain, site is healthy. **Custom domains not impacted** unless genuinely phishing — treat custom-domain trust errors as a separate issue. _Keywords: dangerous site, works in Edge, custom domain unaffected, Google Safe Browsing, contentstackapps.com, trust error._
- [Google Safe Browsing "Dangerous site" flag on default domain](custom-domains-ssl/launch-default-domain-google-safe-browsing-dangerous-site-flag.md) — shared `*.contentstackapps.com` default domains get periodically flagged; platform/DevOps clears it (not customer-fixable). Stopgaps (Details→proceed, Firefox) aren't customer-facing fixes — **use a custom domain** for production. _Keywords: Google Safe Browsing, dangerous site, contentstackapps.com flagged, default domain, support portal, Cloudflare incident._
- [Branded click-tracking reverse proxy (Braze/awstrack.me) + 7 requirements](custom-domains-ssl/launch-branded-click-tracking-reverse-proxy-rewrites-requirements.md) — Launch can reverse-proxy a branded click domain to the tracking endpoint via **`launch.json` rewrites** (`/:path*` → `https://r.us-east-1.awstrack.me/:path*`) or Edge Functions; auto SSL, preserve path/query, pass redirects through, **no caching / no WAF-bot-JS challenges**. Don't ask CS to flip Cloudflare orange-cloud proxy. _Keywords: branded click tracking, Braze, awstrack.me, launch.json rewrite reverse proxy, disable caching, pass through redirects, Cloudflare orange cloud._
- [SES click tracking on a branded custom domain (HTTPS)](custom-domains-ssl/launch-ses-click-tracking-custom-domain-https-proxy.md) — `server: cloudflare` is normal; CNAME direct to SES for HTTP, **Next.js catch-all proxy route** for branded-cert HTTPS. _Keywords: AWS SES, awstrack.me, Braze, cert mismatch, TLS SNI._

## edge-functions-rewrites/ — Edge Functions, Edge URL Rewrites / launch.json

- [Edge Functions in TypeScript](edge-functions-rewrites/edge-functions-typescript.md) — Edge Functions run **JavaScript only**; author in TS, **compile to JS** into the `functions/` folder (sample `tsconfig.json` included). _Keywords: edge function, TypeScript, JavaScript only, tsconfig, build to functions folder._
- [Subdomain→subdomain redirect + Edge Function vs Next.js middleware](edge-functions-rewrites/launch-subdomain-redirect-edge-function-vs-nextjs-middleware.md) — custom-domain UI redirect is **apex→subdomain only**; use an **Edge Function** (302) for subdomain→subdomain. Edge Functions beat Next.js middleware — **Launch doesn't run Next.js middleware at the edge** (server-side only). _Keywords: 302 redirect, apex to subdomain only, Edge Function redirect, Next.js middleware not on edge, personalization, pending domain DCV._
- [Phased migration: rewrite some paths to Launch, rest to legacy (Umbraco)](edge-functions-rewrites/launch-phased-migration-rewrites-legacy-backend.md) — point the **primary domain at Launch**, use **`launch.json` rewrites / Edge Functions** to serve migrated paths from Launch+Contentstack and forward the rest to the legacy origin. Single public domain; no legacy-frontend change. Includes a scoping checklist. _Keywords: phased migration, Umbraco, incremental cutover, split traffic, rewrite to legacy, single domain, launch.json, Edge Functions._
- [launch.json rewrites "lost" after a failed deployment](edge-functions-rewrites/launch-json-rewrites-config-on-failed-deployment.md) — failed deploy does **not** wipe config; last successful config stays live. File must be at **project root**. _Keywords: launch.json, edge rewrites, config lost, Vercel migration._

## deployments-builds/ — deployments, build machines, build/install failures

- [Build fails with no error in log → enable L2 build machine](deployments-builds/launch-build-failed-no-error-l2-build-machine.md) — log stops mid-process = under-provisioned machine; enable `contentfly_deployment_build_machine_l2`. _Keywords: build failed no error, stuck mid-log, L2 build machine, Enterprise default tier._
- [npm E401 "Incorrect or missing password" on install](deployments-builds/launch-build-npm-e401-auth-token-npmrc.md) — private-registry **auth token** missing in build env (works locally via cached creds); fix `.npmrc` + env vars. _Keywords: npm E401, .npmrc, NPM_TOKEN, private registry, fontawesome/GitHub Packages._
- [Blank pages from `.next/cache` ENOENT on read-only FS](deployments-builds/launch-nextjs-cache-enoent-blank-pages-readonly-fs.md) — `/var/task` is read-only; relocate cache to `/tmp` via `cacheHandlers`. ENOENT ≠ ENOSPC. _Keywords: blank pages, .next/cache, ENOENT, read-only filesystem, cacheHandlers._
- [Deployment stuck/queued for minutes → DB replication latency](deployments-builds/launch-deployment-stuck-queued-db-replication-latency.md) — transient **platform incident** (DB migration cleanup → replica load → replication latency); affects only the **Launch UI/management plane**, not production sites. Re-trigger the deploy. _Keywords: deployment stuck, queued, replication latency, DB migration, no production impact, retry._
- [MicroStrategy: queued deploy, Cancel failed, CLI-vs-UI](deployments-builds/launch-microstrategy-deployment-queued-cancel-failed-cli-vs-ui.md) — same DB-latency incident; **"Cancel deployment" can also fail** (redeploy a previous success instead); **CLI would NOT avoid the delay** (backend processing degraded); production traffic safe but deploy *operations* disrupted. _Keywords: force-fail deployment, cancel failed technical error, Launch CLI vs UI, recurring stuck deploy, RCA, mitigation._
- [Deployment queued/failed + Live Preview not working](deployments-builds/launch-deployment-queued-failed-live-preview-not-working.md) — separate the deploy failure from Live Preview (iframe CORS/CSP); check SSO-only access for support. _Keywords: deployment queued, Live Preview SDK not initialized, CORS/CSP iframe._
- [504 after a successful deploy → Azure regional incident](deployments-builds/launch-504-after-deploy-azure-incident-hold-redeploys.md) — redeploy/rollback don't help because the cause is an **upstream Azure NA (US West 2) incident**, not the code; **hold redeploys** during the outage (Launch may manually restore), then redeploy once a **DR mechanism** unblocks it. _Keywords: 504 after deploy, redeploy/rollback no help, Azure US West 2, power event, hold redeploys, disaster recovery._
- [504 on live sites ("working until yesterday") → Azure incident](deployments-builds/launch-504-sites-down-azure-uswest2-incident.md) — both UAT/dev sites 504 with no deploy involved; upstream **Azure NA US West 2 power event** (see Contentstack Status incident); hold redeploys, Launch manually restores, DR mechanism unblocks redeploys. _Keywords: 504, bad gateway, azcontentstackapps.com, working until yesterday, Azure US West 2, power event, status page._
- [Deployment stuck / hangs indefinitely](deployments-builds/launch-deployment-stuck.md) — build command running a **server process** (e.g. `node server.js`) blocks the build step; remove it. Deployments auto-timeout at **60 min**. Standalone Express backend not supported; use Cloud Functions instead. _Keywords: deployment hung, stuck deploy, Express server, SET PORT, build command, timeout._
- [SPA `NoSuchKey` / 404 on direct URL navigation](deployments-builds/launch-spa-nosuchkey-404.md) — Launch can't fall back to `index.html` for unmatched routes; use **HashRouter** as workaround. SPA preset planned. _Keywords: NoSuchKey, 404, single page app, BrowserRouter, HashRouter, React starter._

## caching-cdn/ — CDN caching, invalidation, origin traffic, personalization

- [CDN caching vs personalization; no cache-key variation by cookie](caching-cdn/launch-cdn-caching-personalization-no-cache-variation.md) — Launch caches by **URL only** (no cookie/header Vary); use `no-store` for personalized SSR, CDN-cache only shared data; no `revalidateTag/Path`. _Keywords: personalization, cookie cache, no-store, s-maxage, Revalidate CDN Cache API._
- [Cloudflare APO, cache purge, multi-CDN invalidation](caching-cdn/launch-cloudflare-apo-cache-purge-multi-cdn-invalidation.md) — APO **not** supported/exposed; Launch **doesn't call the Cloudflare API** (use Launch **Cache Revalidation API**, no CF tokens needed); with a CDN in front of Launch, invalidation **doesn't propagate** — purge each layer separately or don't cache upstream. _Keywords: Cloudflare APO, cache purge webhook, Revalidate CDN Cache API, multi-CDN, external CDN, Host header, Contentstack Automate._
- [Cache persists past s-maxage → non-standard `60s` value](caching-cdn/launch-cache-persists-past-s-maxage-nonstandard-format.md) — `s-maxage` must be **integer seconds** (`s-maxage=60`), not `60s`; the unit suffix is non-standard and parsed inconsistently by CDNs → over-long caching / stale content. Fix the header, then re-measure. _Keywords: cache not clearing, drift, s-maxage=60s non-standard, Cache-Control format, stale after publish._
- [Cache key can't use a custom header → per-locale domain / query-param](caching-cdn/launch-cache-key-custom-header-locale-domain-queryparam-workaround.md) — Launch cache keys can't vary by a custom header (e.g. Akamai locale/domain header), so requests collide. To get per-locale caching + `stale-while-revalidate`, put the distinction in the **hostname (domain per locale)** or **URL (`?locale=`)**. _Keywords: cache key custom header, vary by header, Akamai locale header, stale-while-revalidate, domain per locale, locale query param._
- [Sustained origin traffic surge / 404s on unpublished asset](caching-cdn/launch-sustained-origin-traffic-surge-404-unpublished-asset.md) — external crawlers hitting historical URLs (not Launch caching); Tiered Cache ≠ per-edge revalidation; block/observe via Edge Function. _Keywords: traffic surge, 404 unpublished entry, tiered cache, node-fetch, 24/7 non-diurnal._

## logs-monitoring/ — Log Targets, log volume, uptime monitoring

- [Server logs won't load (403/503) + viewer auto-scrolls to bottom](logs-monitoring/launch-server-logs-not-loading-403-503-autoscroll.md) — 403 console/503 screen = platform bug **CL-4060** (fixed in prod); separate UX issue: viewer **auto-scrolls to latest** so you can't read older logs (CL-4123 backlog) — use **Log Targets** for historical analysis. _Keywords: server logs not loading, 403 503, CL-4060, auto-scroll to bottom, log viewer, Log Targets, SSO access._
- [Cold starts after short idle → keep warm with Pingdom](logs-monitoring/launch-cold-starts-idle-container-keep-warm-pingdom.md) — expected: Launch (Azure) **terminates idle containers after ~5 min**, so cold starts after short inactivity are normal (esp. non-prod). Keep a low-traffic prod site warm with a **customer-managed Pingdom uptime check**. _Keywords: cold start, idle container terminated, keep warm, Pingdom, Azure 5 minutes, intermittent traffic._
- [Load-test response-time spike → buffered responses + app-side time](logs-monitoring/launch-load-test-response-time-spike-buffered-responses.md) — not autoscaling/restarts (none found); split latency into **platform-side response buffering** (fixed, re-test) vs **time spent in the application** (customer to optimize). _Keywords: load test, latency spike, autoscaling, infrastructure restart, buffered response, request duration, app-side processing._
- [Estimating Non-Production log volume](logs-monitoring/launch-nonprod-log-volume-estimate-log-targets.md) — Launch can't estimate it (internal format ≠ forwarded, short retention); measure via a **Log Target** in the observability platform. _Keywords: log volume, Log Target, OpenTelemetry, ingestion volume, retention._
- [Brief downtime + "No server logs to display" → upstream Azure incident](logs-monitoring/launch-downtime-no-server-logs-azure-regional-incident.md) — 504s with no app logs = **upstream cloud-provider (Azure US West 2) regional incident**, not a Launch fault; provider recovery window can extend past the status-page "resolved" time. _Keywords: downtime, no server logs, 504 Gateway Timeout, Azure NA incident, status page, recovery window, timezone._
- [Site down "Request URI does not exist" / redirect 404](logs-monitoring/microstrategy-site-down-uri-does-not-exist-redirect-404.md) — customer rebrand/deploy dropped a redirect mapping; not a platform fault; monitor target may need updating. _Keywords: site down, Request URI does not exist, redirect 404, monitor, rebrand._

## cloud-functions/ — Launch Cloud Functions, runtimes, dependencies, routing

- [Node modules / npm dependencies in cloud functions](cloud-functions/launch-cloud-functions-node-modules-dependencies.md) — dependencies via `package.json` **are supported** (Node.js 16 runtime); builds via the **Build command** option; default response cache is `no-cache`, override with `Cache-Control` headers. _Keywords: cloud functions, node_modules, npm, runtime, caching, cache-control._
- [Cloud functions not supported in monorepos](cloud-functions/launch-cloud-functions-monorepo.md) — **Cloud functions don't work in monorepos**; Next.js API routes (`/api`) are the alternative and do work. Edge functions were not yet supported at time of thread. _Keywords: cloud functions, monorepo, Next.js API routes, edge functions._
- [Cloud function fails to deploy with long file path (CFLY-510)](cloud-functions/launch-cf-long-path.md) — **bug**: Lambda deploy fails when the function's folder path is too long; workaround is to shorten the path. _Keywords: cloud function deploy failed, long filepath, Lambda, CFLY-510._

## networking-connectivity/ — egress/outbound IPs, allowlisting, connectivity errors

- [No fixed outbound/egress IPs for allowlisting](networking-connectivity/launch-no-fixed-outbound-egress-ip-allowlist.md) — Launch has **no static egress IPs**; use auth-secured endpoint or a customer proxy/gateway with a static IP. Marketplace App IPs ≠ Launch. _Keywords: outbound IP, egress IP, allowlist, firewall, 403, static IP roadmap._
- [Email 404s on `/cdn-cgi/l/email-protection` (crawler reports)](networking-connectivity/launch-cloudflare-email-obfuscation-cdn-cgi-404-crawler.md) — **Cloudflare Email Obfuscation** (Scrape Shield) rewrites emails to a placeholder URL that 404s for crawlers/direct hits by design; real users unaffected. **Not a Launch change** — ignore in SEO reports. _Keywords: /cdn-cgi/l/email-protection 404, email obfuscation, Scrape Shield, crawler false 404, Screaming Frog, not Launch._
- [Platform limits: deploy-hook secret, log viewer, password-protection OPTIONS](networking-connectivity/launch-platform-limitations-deploy-hook-secret-logs-password-protection-options.md) — deploy hooks have **no native secret** (URL is the secret, like Vercel; use Public APIs/Automate); server-log viewer no-pause/"No logs"/memory issues tracked (**CL-4123**); password protection challenges **all requests incl. `OPTIONS`** at the edge by design — exempt via an **Edge Function**. _Keywords: deploy hook secret, server logs pause, no logs, password protection OPTIONS CORS preflight, Edge Functions, Public API, Automate._
- [Cloudflare Email Protection breaks mailto under strict CSP](networking-connectivity/launch-cloudflare-email-protection-csp-nonce-mailto.md) — Email Protection is **zone-level, can't be disabled per project**; CSP nonces block the injected decode script. Exclude emails with `<!--email_off-->...<!--/email_off-->`; long-term use contact forms. _Keywords: Cloudflare Email Protection, email obfuscation, CSP nonce, mailto broken, email_off, zone-level setting._
- [Imperva: apex→www redirect + whitelisting Imperva IPs in Launch](networking-connectivity/launch-imperva-redirect-apex-www-ip-whitelist-edge-function.md) — apex→www via **Custom Domain / Edge redirects**; Launch has **no IP-allowlist UI** — with no IP restriction configured, nothing blocks the WAF so whitelisting isn't needed; enforce IP rules via an **Edge Function**. _Keywords: Imperva, whitelist IPs, webserver firewall, apex to www redirect, IP-based access control Edge Function._
- [Imperva WAF in front of Launch → Cloudflare Error 1000](networking-connectivity/launch-imperva-waf-cloudflare-error-1000-host-header.md) — nested proxying to default `contentstackapps.com` triggers Error 1000; route to a **custom domain** as origin and set a **Host Header Override** on the WAF (the actual fix). O2O doesn't apply to non-Cloudflare WAFs. _Keywords: Imperva, Cloudflare Error 1000, external WAF, Host header override, nested proxying, O2O._
- [DELETE method with body → CF1003 / 502](networking-connectivity/launch-delete-method-body-cf1003.md) — Launch **doesn't support a payload on GET/DELETE** (RFC 7231); DELETE+body fails upstream before reaching the handler. Move identifiers to **query params**. _Keywords: CF1003, 502, DELETE with body, upstream rejected, RFC 7231._
- [Next.js OAuth authorize/state → 520](networking-connectivity/nextjs-oauth-simple-authorize-state-520.md) — **520** = origin didn't return a valid response (not a WAF block, which is 403); extra redirects + `state`/latency in the OAuth chain caused incomplete responses. _Keywords: OAuth, simple-authorize, state param, 520, redirects, slow origin._
- [Akamai CDN compatibility + CONTENTSTACK_REGION SDK quirk](networking-connectivity/launch-akamai-cdn.md) — Launch works with Akamai (external CDN in front of Launch) with caveats; use region `'us'` not `'NA'` in SDK init—empty string causes invalid CDN hostname. _Keywords: Akamai, external CDN, CONTENTSTACK_REGION, us region, NoSuchKey, CF001._
