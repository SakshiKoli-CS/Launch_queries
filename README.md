## custom-domains-ssl/ — custom domains, apex/A records, DCV, SSL/TLS

- ⭐ **[CANONICAL: contentstackapps.com Dangerous-Site/Malware flag — root cause, RCA, fix](custom-domains-ssl/launch-contentstackapps-dangerous-site-malware-canonical-rca-fix.md)** — Google **false-positive** flag on the shared default domain (triggered by `{brand}-dev` resembling the real brand site), cascaded to all `*.contentstackapps.com`. Fix: verify domain in **Google Search Console** (TXT via Cloudflare) → **Request Review** → add **noindex/robots.txt**. **[Official doc](https://www.contentstack.com/docs/launch/troubleshooting-browser-dangerous-site-warnings)**. - [Cert stuck Pending → DCV needs a CNAME not TXT (+ domain limit, go-live)](custom-domains-ssl/launch-cert-pending-dcv-needs-cname-not-txt-domain-limit-golive.md) — cert/DCV validation needs a **CNAME** (`_acme-challenge… CNAME …dcv.cloudflare.com`), hostname validation needs **TXT** — a TXT in place of the DCV CNAME keeps the cert Pending. - ["Coming Soon" page DNS + project/environment reuse](custom-domains-ssl/launch-coming-soon-page-dns-project-environment-reuse.md) — Launch **doesn't hand out DNS records**; configure a custom domain in the UI and add the **auto-generated** records. ## caching-cdn/ — CDN caching, invalidation, origin traffic, personalization

- [CDN caching vs personalization; no cache-key variation by cookie](caching-cdn/launch-cdn-caching-personalization-no-cache-variation.md) — Launch caches by **URL only** (no cookie/header Vary); use `no-store` for personalized SSR, CDN-cache only shared data; no `revalidateTag/Path`. _Keywords: personalization, cookie cache, no-store, s-maxage, Revalidate CDN Cache API._
- [Cloudflare APO, cache purge, multi-CDN invalidation](caching-cdn/launch-cloudflare-apo-cache-purge-multi-cdn-invalidation.md) — APO **not** supported/exposed; Launch **doesn't call the Cloudflare API** (use Launch **Cache Revalidation API**, no CF tokens needed); with a CDN in front of Launch, invalidation **doesn't propagate** — purge each layer separately or don't cache upstream. _Keywords: Cloudflare APO, cache purge webhook, Revalidate CDN Cache API, multi-CDN, external CDN, Host header, Contentstack Automate._
- [Cache persists past s-maxage → non-standard `60s` value](caching-cdn/launch-cache-persists-past-s-maxage-nonstandard-format.md) — `s-maxage` must be **integer seconds** (`s-maxage=60`), not `60s`; the unit suffix is non-standard and parsed inconsistently by CDNs → over-long caching / stale content. Fix the header, then re-measure. _Keywords: cache not clearing, drift, s-maxage=60s non-standard, Cache-Control format, stale after publish._
- [In-memory LRU cache ineffective (CDA usage) → proxy + CDN caching](caching-cdn/launch-cda-usage-lru-cache-ineffective-proxy-cdn-caching.md) — Launch instances are **short-lived and don't share memory**, so an in-memory LRU (per instance) barely helps and **bigger size won't fix it**; centralize with a **proxy layer + CDN caching** so all instances share cache. Peak mem ~800 MB–1 GB/instance. _Keywords: CDA usage, LRU cache, cache not shared across instances, short-lived instances, proxy CDN caching, cache wrapper, reduce CDA calls._
- [Cache key can't use a custom header → per-locale domain / query-param](caching-cdn/launch-cache-key-custom-header-locale-domain-queryparam-workaround.md) — Launch cache keys can't vary by a custom header (e.g. Akamai locale/domain header), so requests collide. To get per-locale caching + `stale-while-revalidate`, put the distinction in the **hostname (domain per locale)** or **URL (`?locale=`)**. _Keywords: cache key custom header, vary by header, Akamai locale header, stale-while-revalidate, domain per locale, locale query param._
- [Home-page Cache-Control overwritten to no-store → Cloudflare bot cookie BYPASS](caching-cdn/launch-homepage-cache-control-overwritten-cloudflare-bot-cookie-bypass.md) — `/` returns `no-store` + `cf-cache-status: BYPASS` while other pages cache fine; cause is **Cloudflare bot management / `__cf_bm` cookie** forcing a BYPASS (not Next.js — local prod build + `x-cache-control` proved app is correct). Escalate to Cloudflare. _Keywords: cache-control overwritten, homepage no-store, cf-cache-status BYPASS, __cf_bm, bot management, x-cache-control, SWR, API usage spike._
- [Sustained origin traffic surge / 404s on unpublished asset](caching-cdn/launch-sustained-origin-traffic-surge-404-unpublished-asset.md) — external crawlers hitting historical URLs (not Launch caching); Tiered Cache ≠ per-edge revalidation; block/observe via Edge Function. _Keywords: traffic surge, 404 unpublished entry, tiered cache, node-fetch, 24/7 non-diurnal._

- [500s → content-length mismatch + headers-after-sent (app middleware/downstream)](networking-connectivity/launch-500-content-length-mismatch-headers-sent-middleware-downstream.md) — `RequestContentLengthMismatchError` (forwarding a stale `Content-Length` across requests) + `ERR_HTTP_HEADERS_SENT` + downstream `fetch failed`. App-side, not Launch. "Nothing changed" doesn't rule it out — a downstream outage 500s the app. _Keywords: 500, UND_ERR_REQ_CONTENT_LENGTH_MISMATCH, content-length mismatch, ERR_HTTP_HEADERS_SENT, middleware fetch failed, downstream, no code change._
- [Login route hangs → 524/520; chunked 301 redirect missing terminator](networking-connectivity/launch-524-520-login-redirect-chunked-no-terminator-content-length.md) — `/api/auth/signin` 301 has `Transfer-Encoding: chunked` but **no chunk terminator**, so the proxy waits ~98s → 524. Fix: set **`Content-Length: 0`** on redirect (3xx) responses. Works locally (no intermediate proxy); App Router/RSC only. _Keywords: 524, 520, login hangs, NextAuth redirect, chunked no terminator, Content-Length 0, App Router RSC, works locally._
- [DELETE fails CF1003/502 (GET/POST work) → DELETE has a body](networking-connectivity/launch-delete-method-cf1003-502-red-panda-query-params.md) — DELETE with a body isn't supported (RFC 7231), rejected upstream before the handler; "worked until recently" = not portable, not a regression. Move payload to **query params**. _Keywords: CF1003, 502, DELETE with body, GET POST work, RFC 7231, query params, worked until recently._
- [DELETE method with body → CF1003 / 502](networking-connectivity/launch-delete-method-body-cf1003.md) — Launch **doesn't support a payload on GET/DELETE** (RFC 7231); DELETE+body fails upstream before reaching the handler. Move identifiers to **query params**. _Keywords: CF1003, 502, DELETE with body, upstream rejected, RFC 7231._
- [Next.js OAuth authorize/state → 520](networking-connectivity/nextjs-oauth-simple-authorize-state-520.md) — **520** = origin didn't return a valid response (not a WAF block, which is 403); extra redirects + `state`/latency in the OAuth chain caused incomplete responses. "Worked for months, no changes" ≠ platform — prior incidents on this app (CF1003, OPTIONS) were **app-code fixes**. _Keywords: OAuth, simple-authorize, state param, 520, redirects, slow origin, sudden break no change, CF1003, OPTIONS._
- [Akamai CDN compatibility + CONTENTSTACK_REGION SDK quirk](networking-connectivity/launch-akamai-cdn.md) — Launch works with Akamai (external CDN in front of Launch) with caveats; use region `'us'` not `'NA'` in SDK init—empty string causes invalid CDN hostname. _Keywords: Akamai, external CDN, CONTENTSTACK_REGION, us region, NoSuchKey, CF001._
